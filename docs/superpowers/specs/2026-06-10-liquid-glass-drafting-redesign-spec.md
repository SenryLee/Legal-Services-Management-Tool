# Liquid Glass 与文书起草双模式改造 SPEC

## 1. 目标

把当前「法律人业务管理系统」从偏米色台账工具，调整为更明显的 Liquid Glass 桌面工作台；同时重构「文书起草」为两个清晰模式：

- **模板起草**：基于 `.docx` 模板，AI 负责识别变量、追问缺失信息、填充模板并导出 `.docx`。
- **自由起草**：无模板时，AI 基于文书类型、事实、诉求和风格生成完整初稿，并可继续修改、导出 `.docx`。

改造必须保持本地优先、Markdown 工作区和用户确认写入原则不变。

## 2. 非目标

- 不引入账号、云端数据库或协作后端。
- 不把所有业务模块重写为新架构。
- 不把法律判断自动定案；AI 只产出草稿、变量建议和风险提示。
- 不为了视觉效果牺牲台账、表格、长文本的可读性。

## 3. 当前问题

- Liquid Glass 不能只出现在输入舱；导航、按钮、分段控件、选中态、抽屉、弹窗、状态提示都需要统一交互语言。
- 当前主色偏米色/棕黑，和参考图的黑底、强毛玻璃、边缘高光、白色胶囊主按钮差距明显。
- `DocumentDrafter.tsx` 内联样式过多，按钮/气泡/模板库视觉和主应用割裂。
- 模板起草与自由起草混在一个聊天流里，用户不容易理解当前模式、下一步动作和导出结果。
- 自由起草当前只输出聊天文本，和「生成 .docx」的产品承诺不完全一致。

## 4. 视觉方向

采用 **全局强液态玻璃**，参考用户提供图片的特征。该视觉方向适用于整个应用外壳、导航、按钮、分段控件、抽屉、弹窗、文书起草和 AI 相关区域；台账和长文本区域仍使用深色可读底板，但交互状态必须服从同一玻璃规则。

- 深色底或高反差背景承托玻璃层。
- 玻璃层使用 `backdrop-filter: blur(28px-40px) saturate(1.4-1.9)`。
- 玻璃容器有 1px 半透明高光边、内高光、内阴影和外部柔阴影。
- 主要操作按钮用白色高对比胶囊按钮；次级按钮使用半透明深色胶囊。
- 圆角更大：输入舱、抽屉、浮层建议 24-36px；普通按钮 12-18px。
- 避免米色/棕色主导；基础色改为黑蓝、雾白、冰蓝、青绿、少量紫色。

建议 token：

```css
--bg-app: #05080b;
--bg-aurora: #0b1620;
--text-main: #f6f8fb;
--text-muted: rgba(236, 242, 248, 0.68);
--glass-dark: rgba(12, 16, 20, 0.58);
--glass-soft: rgba(255, 255, 255, 0.10);
--glass-line: rgba(255, 255, 255, 0.22);
--accent-blue: #70d7ff;
--accent-mint: #52e2c0;
--accent-action: #ffffff;
--danger: #ff6b5f;
--warning: #e4b85d;
--success: #52e2a8;
```

## 5. 全局交互规则

Liquid Glass 是全局交互系统，不是装饰层。

### 5.1 导航

- 默认：透明或弱玻璃底。
- hover：背景亮度提高，出现轻微内高光。
- selected：深色玻璃胶囊 + 左侧/图标冰蓝高光 + 1px 高光边。
- 禁止只用米色块或普通灰色 hover。

### 5.2 按钮

- primary：白色胶囊按钮，深色文字，hover 时增加外发光和轻微上浮。
- secondary：深色半透明玻璃按钮，白色文字，hover 时边缘更亮。
- icon button：圆形玻璃底，选中时使用青绿或冰蓝小光晕。
- pressed：向内压 1px，内阴影增强。
- disabled：降低透明度，不改变布局。

### 5.3 分段控件与模式切换

- 模板起草 / 自由起草必须是强分段控件。
- 当前模式使用高亮玻璃胶囊，非当前模式保持透明底。
- 切换模式时保留草稿，但必须清晰提示「当前结果属于哪个模式」。

### 5.4 表格与台账

- 表格主体优先可读，不强行全透明。
- 表头可使用轻玻璃；行 hover 使用浅玻璃浮起；选中行使用蓝色高光边。
- 操作列按钮使用圆形玻璃图标按钮。

### 5.5 抽屉、弹窗、状态提示

- 抽屉和弹窗使用强玻璃层，背景加暗色遮罩和 blur。
- 成功、警告、错误提示使用同一玻璃容器，只改变语义色高光。
- AI 生成中使用玻璃层内的流动高光，不使用突兀 loading 文案堆叠。

## 6. 文书起草信息架构

文书起草页拆为三个区域：

- **顶部模式输入舱**：模式切换、输入框、生成按钮、附件/模板入口。
- **左侧上下文面板**：模板变量、事实要素、缺失信息、关联记录。
- **右侧文书预览面板**：草稿正文、修改记录、导出 `.docx`。

模板起草和自由起草共用外观，但状态、提示词和生成结果必须分开。

## 7. 模板起草流程

1. 用户选择已有模板，或上传 `.docx` 转模板。
2. 系统读取模板 metadata 和变量列表。
3. AI 根据用户输入提取变量值。
4. 如果变量缺失，AI 一次只追问最关键的 1-3 个问题。
5. 变量齐备后，系统用 docxtemplater 填充模板并生成 `.docx`。
6. 用户可保存、重新填写变量、或另存为新文书。

模板起草原则：

- 不让 AI 随意改模板原文。
- AI 只负责变量抽取、缺失追问、字段校验和生成说明。
- 若用户要求大幅改写模板，应提示转入自由起草或复制模板后编辑。

### 模板变量收集提示词

```text
你是法律文书模板起草助手。你的任务不是重写模板，而是根据模板变量列表，从用户消息中提取变量值，并判断是否可以填充模板。

规则：
1. 严格保留模板结构和措辞，不得自行增删模板条款。
2. 只提取用户明确提供的信息；不确定则留空。
3. 日期统一 YYYY-MM-DD；金额输出纯数字，可在 note 中说明币种。
4. 一次最多追问 3 个缺失字段，优先追问会阻断生成的字段。
5. 输出严格 JSON，不要 markdown。

输出格式：
{
  "mode": "template",
  "status": "need_more_info | ready",
  "collected": { "placeholder": "value" },
  "missing": [
    { "placeholder": "name", "label": "中文名", "reason": "为什么必需" }
  ],
  "question": "给用户的一句话追问；ready 时为空",
  "risk_notes": ["需要用户复核的法律或事实风险"],
  "user_summary": "给用户看的简短生成说明"
}
```

## 8. 自由起草流程

1. 用户描述文书需求。
2. AI 先判断是否足够起草。
3. 信息不足时，AI 追问关键事实；信息基本足够时直接生成初稿。
4. 初稿进入预览面板。
5. 用户可继续要求修改；系统保留版本记录。
6. 用户可导出 `.docx`，默认使用用户选择的「自由起草默认模板」。
7. 如果用户尚未选择默认模板，系统使用内置基础版式，并引导用户在设置中选择默认模板。

自由起草原则：

- AI 可以生成全文，但必须标出用户未提供的信息占位符。
- 模型不得编造案号、法院、金额、日期、当事人身份。
- 修改时输出完整新版本，不只输出片段。
- 自由起草默认模板只控制 `.docx` 版式，不限制 AI 生成内容；模板中的正文占位符由系统填入 `draft_body`。

### 自由起草默认模板

- 设置页新增「文书起草默认模板」配置项。
- 用户可从已保存模板中选择一个作为自由起草默认模板。
- 默认模板必须至少支持一个正文占位符，例如 `{draft_body}`。
- 可选支持 `{draft_title}`、`{document_type}`、`{generated_date}`、`{risk_notes}`。
- 保存默认模板配置后，仅影响后续自由起草导出，不改变历史草稿。
- 删除某个模板时，如果它是自由起草默认模板，系统应清空该配置并回退到内置基础版式。

### 本地模板文件夹与自动扫描

- 模板库默认绑定当前工作区本地目录：`<workspace>/.legalbiz/templates/docx/`。
- 用户上传、导入或手动放入的 `.docx` 均保存在该目录；模板元数据以同名 `.json` sidecar 文件保存。
- 每次进入文书起草页时，若 `autoScanTemplates` 开启，系统扫描模板目录：
  - 对已有 `.docx + .json` 模板，刷新兼容性状态。
  - 对只有 `.docx` 的新增文件，生成基础元数据。
  - 能识别 `{placeholder}` 的文件标记为「新发现」；包含 `{draft_body}` 的文件标记为「自由起草可用」。
  - 没有占位符的普通 Word 文档标记为「需转换」，不静默调用 AI。
- 模板库 UI 必须提供「打开文件夹」「扫描」「导入」「AI 转模板」入口。
- 设置页提供「自动扫描本地模板文件夹」开关；默认开启。

### 自由起草提示词

```text
你是资深法律文书起草助手。请根据用户需求生成可供律师复核的法律文书草稿。

规则：
1. 如果缺少足以起草的核心信息，先追问，不要强行生成。
2. 如果信息基本足够，直接输出完整文书。
3. 未提供但文书必需的信息，用 [方括号占位] 标注。
4. 不得编造事实、案号、法院、金额、日期或法律关系。
5. 文风应正式、清晰、可修改，避免口语化。
6. 输出严格 JSON，不要 markdown。

输出格式：
{
  "mode": "free",
  "status": "need_more_info | drafted",
  "document_type": "文书类型",
  "questions": ["需要用户补充的问题，最多 3 个"],
  "draft_title": "文书标题",
  "draft_body": "完整文书正文；need_more_info 时为空",
  "assumptions": ["根据用户信息作出的显式假设"],
  "risk_notes": ["需要律师复核的风险点"],
  "next_actions": ["建议用户下一步操作"]
}
```

## 9. 清理范围

必须清理：

- 修复模板 metadata 读取参数错误：`path` 应改为 Tauri 命令需要的 `storedPath`，或新增专用 `readTemplateMetadata` wrapper。
- 移除 `DocumentDrafter.tsx` 底部大量内联样式，迁移到 CSS class 和设计 token。
- 去掉与 Liquid Glass 冲突的普通蓝白按钮样式。
- 整理文书起草里的 emoji 状态文案，改为图标 + 简洁文本。
- 修正自由起草不能导出 `.docx` 的产品缺口。
- 增加自由起草默认模板选择和回退机制。
- 清理无效动态导入警告，避免构建误导。

暂不清理：

- 不重写全部业务模块数据模型。
- 不删除已有 Markdown 工作区兼容逻辑。
- 不移除浏览器 demo 模式。

## 10. 验收标准

### 功能验收

- 模板起草和自由起草是两个明确模式，切换状态清晰。
- 模板起草可完成：选模板 -> 补变量 -> 生成 -> 保存 `.docx`。
- 自由起草可完成：输入需求 -> 生成初稿 -> 修改 -> 保存 `.docx`。
- 用户可选择一个已保存模板作为自由起草默认模板；导出时使用该模板版式。
- 默认模板被删除后，自由起草导出自动回退到内置基础版式，并提示用户重新选择。
- AI 未配置时给出友好引导，不出现红色技术错误条。
- 用户未确认前，不写入业务记录或覆盖模板。

### 视觉验收

- 截图上一眼能看出强 Liquid Glass：深色透光层、背景模糊、边缘高光、软阴影、白色胶囊主按钮。
- 导航选中、按钮 hover/pressed、分段控件选中、表格行 hover、抽屉和状态提示都使用同一玻璃交互规则。
- 全应用外壳采用强 Liquid Glass；表格和长文本使用更实的深色可读底板，但 hover、selected 和操作按钮仍体现玻璃反馈。
- 桌面宽屏和 1080px 最小窗口下无文字溢出、遮挡或按钮挤压。

### 技术验收

- `npm run lint` 通过。
- `npm run build` 通过。
- `cargo test` 通过。
- 新增至少覆盖以下回归点的测试或可执行检查：
  - 模板 metadata 可读取。
  - 模板模式缺变量时返回追问。
  - 自由模式可生成可导出的 docx。
  - 自由起草可使用用户选择的默认模板导出。
  - 默认模板删除后可回退到内置基础版式。
  - 文书起草模式切换不丢失当前草稿。

## 11. 已确认决策

1. 视觉方向采用全局强 Liquid Glass。
2. 自由起草允许用户选择一个「自由起草默认模板」；未选择或模板失效时回退到内置基础版式。
